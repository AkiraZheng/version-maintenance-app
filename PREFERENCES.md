# 用户偏好与工程规范

## ⚠️ AI 主动更新规则

当用户提出新的需求、偏好、或遇到重复出现的问题时，AI 应该主动：

1. **识别偏好**：如果用户多次提到同一类需求，或明确说明某种偏好
2. **更新本文**：将这些偏好记录到 `PREFERENCES.md` 的相应 section
3. **告知用户**：告诉用户"我已将这个偏好更新到 PREFERENCES.md"
4. **后续生效**：之后的对话会自动读取，用户无需重复说明

---

## 开发习惯

### 修改前必确认
- **每次修改前先说明**：要改什么、改在哪里、为什么这样改
- **先说计划再执行**：不直接动手，等用户确认后再改
- **提交前自检**：代码逻辑是否正确、有无遗漏、有无副作用

### 测试与验证
- 修改后先本地测试，确认无问题后再提交
- 提交前用 `git diff` 确认改动内容符合预期
- 需要用户测试的功能，说明测试步骤

## 功能需求偏好

### 弹窗交互
- 切换页面时检查是否有未保存的修改，如有则弹窗提醒
- 防止误操作导致数据丢失（如选择文字时弹窗意外关闭）

### 数据安全
- 重要操作前自动创建备份
- 导入导出功能要能区分备份类型

## 工程结构

- 纯 HTML + CSS + JavaScript，无框架依赖
- 单页应用，所有逻辑在 `app.js`
- 数据存储：localStorage + File System Access API
- 样式：`styles.css`

## Git 提交规范

- commit message 用中文，格式：`feat:` / `fix:` / `docs:` / `refactor:`
- 每次提交聚焦一个功能或修复
- 提交前检查 `git status` 和 `git diff`
